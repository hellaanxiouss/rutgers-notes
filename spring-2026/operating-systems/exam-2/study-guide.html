Chapter Two
Operating System Structures
🛎️
OS Services
What an OS does for you & the system
⌄
📌 Core Idea An OS provides services in two categories: user-helpful services (convenient for users) and system efficiency services (keep the OS running well).

👤 User-Helpful Services:

  • User Interface (UI) — CLI (command line), GUI (graphical), or touch. Every OS has at least one.
  • Program Execution — Load programs into memory, run them, handle normal/abnormal termination.
  • I/O Operations — Running programs may need I/O (files, devices). OS manages this safely.
  • File-System Manipulation — Read/write files, create/delete directories, manage permissions.
  • Communications — Processes exchange info via shared memory or message passing.
  • Error Detection — Detect errors in CPU, memory, I/O devices, user programs and respond appropriately.

⚙️ System Efficiency Services:

  • Resource Allocation — Allocate CPU cycles, memory, I/O devices among multiple concurrent processes.
  • Accounting — Track which users use which resources and how much.
  • Protection & Security — Control access to resources; authenticate users; defend against external attacks.
🎯 Exam Tip

Remember: 6 user-helpful services (UI, Execution, I/O, File-system, Communication, Error Detection) + 3 system efficiency services (Resource allocation, Accounting, Protection). Use mnemonic: U-E-I-F-C-E + RAP

🖥️
User Interfaces
CLI vs GUI vs Touch
⌄
Type Description Examples
CLI Command-Line Interface. Text commands typed directly. Shells can be built into kernel or as system programs. Bash, sh, PowerShell
GUI Desktop metaphor with icons, folders, mouse. Invented at Xerox PARC. More user-friendly. Windows, macOS Aqua, GNOME
Touch Gesture-based input. No mouse. Virtual keyboard. Voice commands. iOS, Android
💡 Key Fact Most modern systems have BOTH CLI and GUI. Mac OS X has Aqua GUI but also supports UNIX shells. Windows has GUI + Command Prompt/PowerShell.
🎯 Exam Tip

CLI: sometimes in kernel, sometimes as program. Adding new commands to shell doesn't require modifying the shell (commands are just program names).

System Calls
The bridge between user programs & OS kernel
⌄
📌 Definition A system call is the programmatic way a user program requests a service from the OS kernel. Think of it as the "API of the OS."
👤 User Program
↓ calls API (e.g., printf)

Parameter Passing Methods:

  • Registers — Simplest: pass params directly in CPU registers (limited by number of registers).
  • Memory Table — Store params in a block/table in memory, pass the address in a register. Used by Linux & Solaris.
  • Stack — Push params onto stack, OS pops them. No limit on number/length of params.

6 Types of System Calls:

File Management
create/delete/open/close/read/write files
Device Management
request/release devices, read/write devices
Information Maintenance
get/set time, date, system data, attributes
Communications
create/delete connections, send/receive messages
Protection
control access, get/set permissions
🎯 Exam Tip

Mnemonic for 6 types: P-F-D-I-C-P = Please Find Dave, I Can't Proceed. The caller doesn't need to know HOW system calls work — they just use the API.

🔧
System Programs
Programs that come with the OS
⌄
📌 Core Idea System programs provide a convenient environment for program development and execution. They are the USER's view of the OS — most users see system programs, not system calls.
  • File Management — Create, delete, copy, rename, print, list files and directories.
  • Status Information — Date, time, memory/disk usage, performance metrics. Some use a registry.
  • File Modification — Text editors to create/modify files; commands to search or transform text.
  • Programming Language Support — Compilers, assemblers, debuggers, interpreters.
  • Program Loading & Execution — Loaders, linkage editors, overlay loaders.
  • Communications — Virtual connections between processes, users, systems (email, web, login).
  • Background Services (Daemons) — Launch at boot. Disk checking, process scheduling, error logging, printing. Run in USER context, not kernel.
⚠️ System vs Application Programs System programs help OTHER systems/programs run. Application programs (email clients, browsers, games) serve END USERS directly. Application programs are NOT part of the OS.
🏗️
OS Structure
Simple, Layered, Microkernel, Modular
⌄

Key Design Principle: Policy vs Mechanism

Policy = WHAT will be done? (e.g., which process runs next?)
Mechanism = HOW to do it? (e.g., timer interrupt)

Separating them allows flexibility — policy can change without changing mechanism.
Structure Description Pros/Cons
Simple (MS-DOS) No clear separation of layers. Most functionality in least space. Interfaces not well-separated. ❌ Not modular, insecure, fragile
Monolithic (UNIX) Kernel does everything between hardware and system-call interface. Large but functional. ✅ Fast ❌ Hard to maintain
Layered OS divided into N layers. Layer 0 = hardware, Layer N = UI. Each layer uses only layers below it. ✅ Modular ❌ Performance overhead
Microkernel Moves as much as possible from kernel to user space. Kernel is tiny. Communication via message passing. ✅ Reliable, secure, portable ❌ Performance overhead
Modular Loadable kernel modules. Object-oriented. Each component is separate but interfaces with others. Linux, Solaris. ✅ Flexible, like layers but better
Hybrid Mix of approaches. macOS = Mach microkernel + BSD Unix. Windows = mostly monolithic + microkernel elements. ✅ Best of all worlds
🎯 Exam Tip

Microkernel benefits: easier to extend, easier to port, more reliable (less kernel code), more secure. Main detriment: performance overhead of user↔kernel communication.

🚀
System Boot & Debugging
From power-on to running OS
⌄

Boot Sequence:

1. Power on → CPU starts execution at fixed memory location (ROM)
2. Firmware (BIOS/UEFI) in ROM runs initial boot code
3. Bootstrap loader (GRUB) found in ROM/EEPROM — locates kernel on disk, loads it into memory
4. Kernel is loaded and system starts running 🎉
Debugging Terms Core dump = memory snapshot when application crashes
Crash dump = kernel memory snapshot when OS fails
Profiling = sampling instruction pointer to find performance hotspots
DTrace = Solaris tool for live system instrumentation
SYSGEN SYSGEN (System Generation) configures the OS for specific hardware. Can produce more efficient code than a general-purpose kernel.

📋
Process Control Block (PCB)
The OS's data structure for each process
⌄
📌 What is PCB? The PCB (also called Task Control Block) is a data structure the OS maintains for each process. It contains ALL information about a process. When the OS switches processes, it saves/restores PCB data.
📋 PCB
Process State (new/ready/running/...)
Program Counter (next instruction)
CPU Registers (all register values)
CPU Scheduling Info (priorities, queue pointers)
Memory Management Info
Accounting Info (CPU time used, time limits)
I/O Status Info (devices allocated, open files)
🎯 Exam Tip

Mnemonic: S-P-C-S-M-A-I = "Some Programmers Can Save Many Apps Indeed" → State, Program counter, CPU registers, Scheduling info, Memory info, Accounting info, I/O info.

📋
Schedulers & Queues
Short-term, Long-term, Medium-term
⌄
Scheduler Also Called Frequency Job
Short-term CPU Scheduler Very frequent (ms) Selects which READY process gets the CPU next. Must be FAST.
Long-term Job Scheduler Infrequent (seconds/min) Selects which processes enter the ready queue from disk. Controls degree of multiprogramming. Can be slow.
Medium-term Swapper As needed Removes processes from memory to disk (swapping) to reduce multiprogramming degree temporarily.

Process Types (for Long-term scheduler):

⚡ I/O-Bound Process

Spends more time doing I/O. Many SHORT CPU bursts. Example: file editor, web browser.

🔥 CPU-Bound Process

Spends more time computing. Few LONG CPU bursts. Example: scientific calculations, video encoding.

Long-term scheduler aims for a good mix of I/O-bound and CPU-bound processes for optimal system performance.
🎯 Exam Tip

Three queues: Job Queue (all processes in system), Ready Queue (in memory, ready to run), Device Queue (waiting for I/O device). Processes migrate between these queues constantly.

🌱
Process Creation & Termination
fork(), exec(), exit(), zombie & orphan
⌄
Processes form a tree. Parent creates children using system calls. Each process has a unique PID (Process Identifier).

UNIX Process Creation:

  • fork() — Creates a new child process. Child is a DUPLICATE of parent. Returns PID to parent, 0 to child.
  • exec() — Called after fork(). Replaces the process's memory space with a new program. Child becomes a different program.
  • wait() — Parent waits for child to terminate. Returns child PID and status.
  • exit() — Process terminates itself, returns status to parent.
🧟
Zombie Process
click to reveal
Process has finished (exit()) but parent hasn't called wait() yet. PCB still exists to store exit status.
👶
Orphan Process
click to reveal
Parent terminated without calling wait(). Child continues running without a parent. Usually re-parented to init.
🌊
Cascading Termination
click to reveal
When parent terminates, OS terminates ALL its children too (grandchildren, etc.). Initiated by OS.
🎯 Exam Tip

Chrome browser = 3 process types: Browser process (UI, I/O), Renderer process (one per website, sandboxed), Plugin process (one per plugin). This is why one tab crashing doesn't kill the whole browser!

💬
Interprocess Communication (IPC)
Shared Memory vs Message Passing
⌄
Cooperating processes can affect/be affected by each other. They need IPC. Reasons: information sharing, computation speedup, modularity, convenience.
🧠 Shared Memory
  • Processes share a region of memory
  • Under control of USER processes (not OS)
  • Must synchronize access!
  • Faster — no kernel involvement after setup
  • Producer-Consumer model: shared buffer
📨 Message Passing
  • send(message) / receive(message)
  • OS manages the communication link
  • Easier to implement, no synchronization needed
  • Slower — every msg goes through kernel
  • Good for distributed systems

Message Passing Details:

  • Direct — Processes name each other explicitly: send(P, msg), receive(Q, msg)
  • Indirect (Mailboxes/Ports) — Messages go to a mailbox with unique ID. Many processes can share a mailbox.
  • Blocking (Synchronous) — Sender/receiver blocks until message is sent/received. Both blocking = rendezvous.
  • Non-blocking (Asynchronous) — Sender sends and continues. Receiver gets valid message or null.

Client-Server Communication:

Sockets — Endpoint = IP:Port. Below port 1024 = well-known services. 127.0.0.1 = loopback.
RPC — Remote Procedure Call. Stubs handle marshalling/unmarshalling params across network.
Pipes — Ordinary: unidirectional, parent-child only. Named: bidirectional, no relationship needed.
RMI — Java's Remote Method Invocation. Similar to RPC but for Java objects.

4 Benefits of Threads (RRES):

⚡
Responsiveness
Continue executing if part is blocked. Critical for UIs.
🔗
Resource Sharing
Threads share process resources automatically. Easier than shared memory or message passing.
💰
Economy
Cheaper than process creation. Thread switching has lower overhead than context switching.
📈
Scalability
Can take advantage of multiprocessor/multicore architectures.
🎯 Exam Tip

Thread creation is light-weight; process creation is heavy-weight. Kernels themselves are generally multithreaded. Per thread: own stack, registers, PC. Shared: code, data, files.

🔲
Multicore Programming
Concurrency vs Parallelism + Amdahl's Law
⌄
⚡ Concurrency

More than one task making progress. Works on a SINGLE core using rapid switching. Gives illusion of simultaneous execution.

T1 ██ T2 ██ T1 ██ T2
← Single Core →
🔀 Parallelism

Truly running more than one task simultaneously. Requires MULTIPLE cores. Real simultaneous execution.

Core 1: T1 ████████████
Core 2: T2 ████████████

Types of Parallelism:

  • Data Parallelism — Same operation on subsets of data across multiple cores (e.g., each core processes part of an array).
  • Task Parallelism — Different threads perform different operations across cores (e.g., one thread downloads, another decodes).
📐 Amdahl's Law Speedup = 1 / (S + (1-S)/N) where S = serial portion, N = cores.

Even if 75% of app is parallel, going from 1→2 cores gives only 1.6x speedup. The serial portion dominates as N→∞, speedup approaches 1/S.

Challenges of multicore programming: Dividing activities, Balance (equal work per core), Data splitting, Data dependency, Testing & debugging. Remember these 5 challenges!

🤖
Implicit Threading
Thread Pools, OpenMP, Grand Central Dispatch
⌄
Implicit threading = compilers/runtime libraries manage thread creation instead of programmers. Growing in popularity as programs get more complex.
  • Thread Pools — Create N threads at startup, put them in a pool awaiting work.
    ✅ Faster (no creation overhead), bounded (pool size), separates task from mechanics. Tasks can be scheduled periodically.
  • OpenMP — Set of compiler directives for C/C++/FORTRAN. Developer marks parallel regions with #pragma. Creates as many threads as there are cores. Handles shared-memory parallel programming.
  • Grand Central Dispatch (GCD) — Apple's technology for macOS/iOS. Blocks of code placed in dispatch queues. Serial queues (FIFO, one at a time) or Concurrent queues (FIFO but multiple simultaneously). Thread pool handles execution.
⚠️
Threading Issues
fork, signals, cancellation, TLS
⌄
  • fork() in multithreaded programs — Does fork() duplicate ALL threads or just the calling thread? Some UNIX systems have two versions. exec() replaces ALL threads with new program.
  • Signal Handling — Signals notify a process of an event. In multithreaded: deliver to the thread it applies to? All threads? A specific thread? Per-process assignment is common solution.
  • Thread Cancellation — Terminating a thread before it finishes.
    Asynchronous: kills immediately (risky!)
    Deferred: thread periodically checks a cancellation flag at safe points (safer, default in Pthreads)
  • Thread-Local Storage (TLS) — Each thread has its OWN copy of certain data. Different from local variables (visible only in one function call). Similar to static data but unique per thread. Useful with thread pools.
  • Scheduler Activations / LWP — Lightweight Process (LWP) = virtual processor between user and kernel threads. For M:M model. Kernel communicates with thread library via upcalls.
🎯 Exam Tip

Linux doesn't distinguish threads from processes — both are "tasks." Thread creation uses clone() system call which allows sharing address space. Windows: ETHREAD (executive), KTHREAD (kernel), TEB (user space).

CPU Scheduling
⏱️
Basic Concepts
CPU burst cycle, Dispatcher, Preemption
⌄
📌 Core Idea Maximize CPU utilization via multiprogramming. When one process waits for I/O, CPU switches to another. This is the basis of all modern OS scheduling.

CPU–I/O Burst Cycle:

CPU Burst
(compute)
I/O Burst
(wait)
CPU Burst
(compute)
I/O Burst
(wait)
...exit
The Dispatcher The dispatcher gives control of CPU to the selected process. It handles: switching context, switching to user mode, jumping to the correct program location.

Dispatch latency = time to stop one process and start another. Should be minimal.

When does scheduling happen?

  • Running → Waiting (I/O request) — Non-preemptive
  • Running → Ready (interrupt) — Preemptive
  • Waiting → Ready (I/O complete) — Preemptive
  • Process Terminates — Non-preemptive
Non-preemptive = process keeps CPU until it voluntarily gives up (done or waiting)
Preemptive = OS can forcibly take CPU away from process (most modern OSes)
📊
Scheduling Criteria
How we measure scheduling performance
⌄
🔥
CPU Utilization
Keep CPU as busy as possible. Goal: MAXIMIZE (typically 40-90%).
📦
Throughput
Number of processes completing per time unit. Goal: MAXIMIZE.
⏰
Turnaround Time
Total time from submission to completion. Goal: MINIMIZE. = finish - arrival.
⌛
Waiting Time
Time spent in the ready queue. Goal: MINIMIZE.
💨
Response Time
Time from request submission to first response (not completion). Goal: MINIMIZE. Key for interactive systems.
🎯 Exam Tip

MAXIMIZE: CPU Utilization, Throughput. MINIMIZE: Turnaround time, Waiting time, Response time. Mnemonic: "Max CPU-T, Min T-W-R". Waiting time = turnaround - burst time.

🚶
FCFS — First Come, First Served
Simplest algorithm — and its problem
⌄
Processes get CPU in the order they arrive. Non-preemptive — once a process starts, it runs until done or waiting. Implemented with a simple FIFO queue.

Example: P1(24ms), P2(3ms), P3(3ms) arriving in order P1, P2, P3:

⚠️ Convoy Effect Short processes get stuck behind one long process, causing everyone to wait. This is the main problem with FCFS. It's like being behind a slow truck in traffic!
⚡
SJF & SRTF
Shortest Job First + preemptive version
⌄
📌 Key Fact SJF is provably optimal — gives minimum average waiting time for a given set of processes. Schedule shortest CPU burst next.
Non-Preemptive SJF

Once a process starts running, it can't be interrupted even if a shorter process arrives.

SRTF (Preemptive)

If a new process arrives with shorter remaining time than current, CPU switches immediately. "Shortest Remaining Time First."

SRTF Example: P1(8ms,arr:0), P2(4ms,arr:1), P3(9ms,arr:2), P4(5ms,arr:3)

👑
Priority Scheduling
Starvation problem & Aging solution
⌄
Each process gets a priority number. CPU allocated to highest priority process. SJF is a special case of priority scheduling (priority = inverse of burst time).
Preemptive

If a higher-priority process arrives, it immediately takes the CPU.

Non-Preemptive

Current process finishes, then highest priority gets CPU next.

😵
Starvation
tap to reveal
Low-priority processes may NEVER execute because high-priority processes keep arriving. They wait indefinitely.
🌱
Aging
tap to reveal
Solution to starvation: gradually increase the priority of waiting processes over time. Eventually they'll run!
🎯 Exam Tip

IMPORTANT: "Highest priority" = SMALLEST priority number (priority 1 > priority 5). Problem = Starvation. Solution = Aging. SJF = priority scheduling with priority = 1/burst_time.

🔁
Round Robin (RR)
Fair time-sharing with time quantum
⌄
📌 Key Idea Each process gets a small CPU time slice (quantum q, typically 10–100ms). When it expires, process is preempted and added to end of ready queue. This is the standard for time-sharing systems.

With n processes and quantum q: each gets 1/n of CPU in chunks ≤ q. Max wait = (n-1)×q.

Example: P1(24ms), P2(3ms), P3(3ms) with q=4:

RR typically has higher average turnaround than SJF, but better response time. Context switch time is typically <10μs. Quantum should be 10–100ms. Rule of thumb: 80% of CPU bursts should be shorter than q.

🏛️
Multilevel & Feedback Queues
The most sophisticated approach
⌄
Multilevel Queue Ready queue split into multiple queues (e.g., foreground/interactive vs background/batch). Each queue has its own algorithm. Process is permanently assigned to a queue.
Priority 1: System Processes ← always runs first
Priority 2: Interactive Processes (RR)
Priority 3: Interactive Editing (RR)
Priority 4: Batch Processes (FCFS) ← 80% of CPU
Priority 5: Student Processes (FCFS) ← whatever's left
Multilevel FEEDBACK Queue Processes can MOVE BETWEEN queues! If a process uses too much CPU time, it gets moved to a lower-priority queue. I/O-bound and interactive processes stay in higher-priority queues. Aging implemented via queue upgrades.

Classic 3-Queue Feedback Example:

Q0: RR with q=8ms → if not done, move to Q1
Q1: RR with q=16ms → if not done, move to Q2
Q2: FCFS (run until done)
🎯 Exam Tip

Multilevel Feedback Queue is the MOST COMPLEX but MOST FLEXIBLE scheduler. Parameters: number of queues, algorithm per queue, when to upgrade/demote, which queue new processes enter. It can implement aging by moving waiting processes to higher queues.

🌐
Multiple-Processor Scheduling
SMP, load balancing, affinity
⌄
  • Asymmetric Multiprocessing — One master processor handles all scheduling. Simple but bottleneck at master.
  • Symmetric Multiprocessing (SMP) — Each processor self-schedules. Most common. Either shared ready queue or private per-CPU queue.
  • Processor Affinity — Process has affinity for CPU it's running on (cache warming). Soft affinity: OS tries but doesn't guarantee. Hard affinity: process specifies which CPUs it can use.
  • Load Balancing — Keep all CPUs busy. Push migration: task checks load and pushes from overloaded CPU. Pull migration: idle CPU pulls tasks from busy CPU.
  • NUMA — Non-Uniform Memory Access. Memory closer to some CPUs. Scheduler considers memory placement for affinity.
Multicore Processors: Multiple cores on single chip. Faster, less power. Can have multiple hardware threads per core. Memory stalls (waiting for memory fetch) allow other threads to run — this is "chip multithreading."
⏰
Real-Time Scheduling
Hard vs Soft, EDF, Rate Monotonic
⌄
Soft Real-Time

Critical processes get priority but NO guarantee of meeting deadline. Best effort.

Hard Real-Time

Task MUST be serviced by its deadline. Missing deadline = system failure. Used in medical devices, aircraft.

  • Interrupt Latency — Time from interrupt arrival to start of service routine. Minimize this!
  • Rate Monotonic — Priority based on PERIOD. Shorter period → higher priority. P1 period=50ms gets higher priority than P2 period=100ms.
  • Earliest Deadline First (EDF) — Dynamically assign priority based on deadline. Earlier deadline → higher priority. Optimal for uniprocessors but complex.
  • Proportional Share — T shares distributed. Each process gets N shares where N/T = fraction of CPU time.
🎯 Exam Tip

Real-time process has: processing time t, deadline d, period p. Relationship: 0 ≤ t ≤ d ≤ p. Rate = 1/p. Rate Monotonic = static priority (based on period). EDF = dynamic priority (based on deadline). EDF can achieve 100% utilization.
